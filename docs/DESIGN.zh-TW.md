# 設計決策記錄 (Design Decisions)

> 這份文件記錄 Compass 從一份「個人開發 SOP」變成「通用 skill」的關鍵設計決策與取捨。
> 作品集的價值不在「做出了什麼」，而在「為什麼這樣設計」——以下是思考過程。
> 涵蓋邊界與未來方向另見 [SCOPE](./SCOPE.zh-TW.md)；這份只談「為什麼」。

---

## 起點：從私人 SOP 到通用 skill

Compass 的原料不是憑空想的，而是一份我自己長期在用的「PRD 驅動開發 SOP」——加上寫在個人 `CLAUDE.md` 裡的「PRD 十二誡」。第一個關鍵體悟是：

> 一份**為自己量身打造**的 SOP，要變成**別人也能用**的 skill，本質不是「複製貼上」，而是**去私有化**。

我的 SOP 裡塞滿了只對我成立的東西：寫死的 FastAPI 結構、特定專案的 PII 模組順序、`~/.claude/開發SOP/` 的私人路徑、我自己寫的 `audit_prd_vs_code.py`。這些對我有用，但對拿到 skill 的陌生人全是雜訊甚至死連結。

**這個體悟定調了整個專案：Compass 的核心工作之一，是把「對我成立」翻譯成「對任何工程師成立」——保留紀律的骨架，抽掉個人的血肉。**

---

## 決策 1：定位為 Sentinel 的「配對 skill」，而非合併成一個

**選項**：(A) 把 PRD 紀律塞進 Sentinel，變一個大 skill　(B) 獨立成第二個 skill，跟 Sentinel 配對

**選擇 B。** 理由：Sentinel 管的是「**怎麼想**」（淺層 vs 深層、根因 vs 症狀），是任何工程任務都適用的思考層；Compass 管的是「**怎麼照規格執行**」，只在「有一份 PRD/spec」時才適用。兩者的**觸發條件不同、適用範圍不同**——硬塞成一個，會讓「改個 typo」也被 PRD 紀律纏住，違反 Sentinel「思考力道隨任務縮放」的精神。

**衍生**：兩個 skill 共用 `.claude/` 病歷檔，但 Compass 的案例加 `[COMPASS]` 前綴方便檢索。一個看思考、一個看規格，合起來才是完整的「動手前 + 動手中」工具組。

> **後記（工具鏈演進）**：本決策當時是兩件式（Sentinel + Compass）；後來加上游 [Cartographer](https://github.com/RayLi-Git/cartographer)（生 PRD），再加下游 [Lookout](https://github.com/RayLi-Git/lookout)（段落完成的獨立審查），形成四件式工具鏈：**Cartographer 畫地圖 → Compass 照圖走 → Sentinel 站哨 → Lookout 瞭望**。「獨立成 skill、各司其職、按需觸發」這條原則對四件同樣成立。

---

## 決策 2：DoR 是 DoD 的對偶——補上「入口閘門」

**體悟**：原 SOP 只有 DoD（Definition of Done，出口閘門），沒有 DoR（Definition of Ready，入口閘門）。但實務上最痛的不是「做完沒驗收」，而是「**PRD 本身就不完整，卻照著它寫了三天**」。

**模糊的 PRD 不會主動跳出來提醒你它模糊**——它在你寫到第三個檔案、要拼一個你以為早就定義好的回傳格式時才爆。那時你已經做了一堆假設、寫了一堆耦合在這些假設上的 code。

**解法**：加一個 §2 Definition of Ready——動手寫第一行 code 前，先健檢 PRD 本身「能不能被實作」（逐端點 method/path/auth/schema/錯誤碼/冪等、逐表約束、認證歸屬…）。**DoR 在入口、DoD 在出口，兩者對偶。** 一個壞 PRD 會無聲地毒化下游所有東西，DoR 就是在源頭攔它。

**關鍵紀律**：DoR 不是「拒絕不完美的 PRD」——真實 PRD 幾乎都不完美。DoR 是「**辨識哪些不完美會擋路、哪些不會**」：有 blocker 退回，有小洞標假設繼續。

---

## 決策 3：最核心的洞見——靠工具強制，不靠紀律

這是 Compass 最有原創價值、也是從實戰學到最深的一條（對應原 SOP §14、Compass §11 的 M-007~M-010）。

**問題**：再多「紀律」「規則」「下次注意」，都擋不住趕工時的自我合理化。「12 個 endpoint 寫完 9 個就打勾」「commit 訊息寫『全部完成』但其實漏一半」——這些不是因為不懂規則，是因為**規則依賴人去記、去自律**。

**解法**：把判斷從「靠記憶」變成「**機械化強制**」：
- **M-007 反聚合**：checklist 禁止「12 個 ⬜」這種聚合條目，必須一行一個 checkbox——漏項無處可藏。
- **M-008 反向稽核**：跑一個 script 比對「PRD 列了什麼 vs code 實作了什麼」，**exit code 非 0 = 失敗**，不靠紀律靠退出碼。
- **M-010 commit 禁主觀詞**：git hook 擋下「完整/done/全部」這種無法驗證的字，逼你用「12/12 endpoints」這種具體計數。

**教訓**：**「靠 exit code 不靠紀律」**——機械化檢查存在卻不跑，等於白建；但只要建了，它每次都會抓，不會因為你累了就放水。這條貫穿整個 Compass 的工具設計。

---

## 決策 4：腳本必須 config-driven，不能綁死我的專案

**矛盾**：我原本的 `audit_prd_vs_code.py` 寫死了 FastAPI 的 `@app.get` pattern、特定的 PRD 章節編號。對我能跑，對別人全錯。

**解法**：把腳本改成 **config-driven**——讀一個 `compass-audit.json`，裡面描述「你的 PRD 在哪、用什麼 regex 抓項目、掃哪些 code 檔」。**腳本本身專案無關**，每個專案自己寫 config。找不到 config 時印出範例並 exit 2（這是設計，不是故障）。

**衍生的教訓**（寫進病歷）：這些印 emoji 的腳本在繁體中文 Windows（cp950 console）會 `UnicodeEncodeError` 崩潰，而崩潰的 exit 1 會跟「PRD 有缺口」的業務碼撞號——靜默誤判。解法是腳本一律 `sys.stdout.reconfigure(encoding="utf-8")`。**通用化不只是抽掉路徑，還要扛得住不同環境。**

---

## 決策 5：「完成就是完成」——與敏捷和解，但不放棄反半成品

**原 SOP 寫**：「一次寫完，不分階段，不切 V1/V2/V3」。但這跟業界主流（Agile / MVP / 漸進交付）直接衝突——漸進交付不是偷懶，是降低風險、拿真實回饋的正當手段。

**體悟**：原 SOP 真正想說的不是「不准分階段」，而是「**不要留半成品**」——不要寫一半跑去寫別的、不要承諾「V2 再補」結果沒 V2。

**修正**：把「一次寫完不分階段」軟化成「**完成就是完成，不留半成品；可分小塊交付，但不可半成品階段化**」。保留反半成品的精神，同時不跟漸進交付對立。關鍵字從「階段」變成「**半成品**」。

**教訓**：把「對我成立」變「對別人成立」時，要分清楚一條規則的**精神**和它的**字面**——字面可能跟別人的工作方式衝突，但精神往往是共通的。

---

## 決策 6：PRD 衝突分軌處理——靜態三類 + 動態三類

**體悟**：「PRD 跟我預期的不一樣」其實是好幾件不同的事，混在一起處理會災難。

**靜態三類**（PRD 不動，你讀它時發現問題）：
- **模糊**：兩種解讀都能做 → 採具體一方，**不停手**。
- **Bug**：自相矛盾、任一解讀都做不出來 → **停手等裁決**。
- **缺漏 + 實作更好**：PRD 沒寫但你自然補了個對齊原則的小東西 → 6 道 gate 檢查，**保留並等裁決**（不直接套 YAGNI 砍）。

**動態三類**（原 SOP 沒涵蓋、但最高頻的真實衝突）：
- **PRD 中途變更**：寫到一半 PRD 升版——這是最痛的，因為已完成的模組可能默默變錯。
- **跨文件衝突**：PRD vs ADR vs OpenAPI vs ERD 互相矛盾，按「誰的領域」定權威。
- **多 PRD 依賴**：多個子 PRD 之間的順序與共享契約。

**關鍵設計**：把「該停的」和「不該停的」精準分流——讓你不會在模糊處浪費時間問人，也不會在真 bug 上硬幹。

---

## 決策 7：既有 code 是「第二份文件」——brownfield 也要紀律

**原 SOP 把 brownfield / 純 bug 修復列為「不適用」。** 但大部分真實工作就是 brownfield。這是個缺口。

**核心 framing**：在既有 codebase 裡，**既有 code 就是 PRD 之外的第二份真相來源**——它記錄了「系統現在實際怎麼跑」，其中可能藏著沒寫進 PRD 的隱性需求、邊界補丁、下游依賴。

**衍生**：當 PRD 跟既有 code 衝突，這不是「PRD bug」，而是**跨文件衝突**（§5.3）——既有 code 是一份文件、PRD 是另一份，按領域定權威，不要盲目照 PRD 覆寫。**風險不對稱**：在 brownfield，弄壞既有行為的代價，遠高於漏做一個新功能。

---

## 決策 8：一份專門的 SCOPE——「說不」也是一種設計

最後一個體悟：一個工具的價值，不只在「它能做什麼」，更在「它清楚說自己**不做什麼**」。

Compass 的 scope 比 Sentinel 複雜得多——「PRD 紀律」和「不是 PRD 紀律」的邊界很容易模糊（PRD 撰寫？產品探索？PM 工具？架構決策？）。所以特地寫了一份獨立的 [SCOPE](./SCOPE.zh-TW.md)，明文列出涵蓋與不涵蓋，避免使用者誤用或期待錯位。

**這也是跟 Sentinel 的分工宣言**：純探索性原型、純改文案、思考方式——那些是 Sentinel 的範圍，不是 Compass 的。一個工具想「什麼都做」，最後會「什麼都做不精」。

## 整體架構哲學

| 層 | 角色 | 對應 |
|---|---|---|
| `SKILL.md` | 常駐指路 + 觸發協定 | 門口木牌（精簡，按需載入 references）|
| `references/` | 11 個主題模組的詳細紀律 | 圖書館書庫（要用才進）|
| `scripts/` + `templates/` | 工具強制的可執行範例 + 追蹤文件範本 | 工具箱（機械化強制的實體）|

核心原則跟 Sentinel 一致：**「指路常駐、細節按需」**——SKILL.md 保持精簡（description 還受 1024 字元上限約束），確保觸發；真正的紀律細節按需載入對應模組，不撐爆 context。

---

> Compass 不是萬靈丹，是一把**特定情境下精準的工具**。同一工具鏈裡，[Cartographer](https://github.com/RayLi-Git/cartographer) 探索「把模糊想法逼成可驗收的 PRD」、[Sentinel](https://github.com/RayLi-Git/sentinel) 探索「把結構化思考編碼成 AI 寫程式的夥伴」；Compass 探索的是「如何把 PRD 驅動開發的紀律，編碼成那個夥伴的執行紀律」。
