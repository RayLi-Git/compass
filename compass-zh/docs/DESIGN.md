# 設計

> 作品集的價值不在「做出了什麼」，而在「為什麼這樣設計」。本文件記錄 Compass 從一份私人開發 SOP 變成通用 skill 的設計理念與關鍵決策。
> 涵蓋邊界與未來方向見 [SCOPE](./SCOPE.md)；這份只談「為什麼」。

---

## 設計理念

Compass 的原料不是憑空想的，而是一份長期私下在用的「PRD 驅動開發 SOP」——加上一份「PRD 十二誡」清單。第一個關鍵體悟定調了整個專案：

> 一份**為一個人量身打造**的 SOP，要變成**任何人都能用**的 skill，本質不是「複製貼上」，而是**去私有化**。

原本的 SOP 裡塞滿了只對作者成立的東西：寫死的 FastAPI 結構、某個專案的 PII 模組順序、一條私人 skills 路徑、一支自製稽核腳本。這些對作者有用，但對拿到 skill 的陌生人全是雜訊，甚至是死連結。

**所以 Compass 的核心工作之一，是把「對我成立」翻譯成「對任何工程師成立」——保留紀律的骨架，抽掉個人的血肉。** 由此推出四個原則：

1. **PRD 是合約，不是建議** — 任何偏差都要記、要對齊、要裁決。
2. **完成就是完成** — 想分小塊交付沒問題，但絕不留半成品階段。
3. **靠 exit code 不靠紀律** — 機械化檢查遠勝規則陳述。
4. **Brownfield 也要紀律** — PRD 紀律不只適用於全新專案。

整體架構與工具鏈夥伴一致：**指路常駐、細節按需。** `SKILL.md` 保持精簡（其 description 還受 ~1024 字元上限約束）以確保觸發，真正的紀律細節按需從 `references/` 逐模組載入——絕不撐爆 context。

| 層 | 角色 | 對應 |
|---|---|---|
| `SKILL.md` | 常駐指路 + 觸發協定 | 門口木牌（精簡，按需載入 references） |
| `references/` | 11 個主題模組的詳細紀律 | 圖書館書庫（要用才進） |
| `scripts/` + `templates/` | 工具強制的可執行範例 + 追蹤文件範本 | 工具箱（機械化強制的實體） |

## 關鍵決策

### 決策 1：獨立成一個 skill，而非合併成一個

- **問題**：該把 PRD 紀律跟思考層塞進一個大 skill，還是獨立拆開？
- **選擇**：拆開。思考層管的是「怎麼想」（淺層 vs 深層、根因 vs 症狀），任何工程任務都適用；Compass 管的是「怎麼照規格執行」，只在有一份 PRD/spec 時才適用。兩者**觸發條件不同、適用範圍不同**。
- **取捨**：要裝兩個 skill 而非一個，而且兩者得共用狀態（`.claude/` 病歷檔，Compass 案例加 `[COMPASS]` 前綴方便檢索）。換來的是：改個 typo 永遠不會被 PRD 紀律纏住。「獨立成 skill、各司其職、按需觸發」這條原則後來延伸成完整的四件式工具鏈。

### 決策 2：DoR 是 DoD 的對偶——補上「入口閘門」

- **問題**：原 SOP 只有 DoD（出口閘門）。但實務上最痛的不是「做完沒驗收」，而是「**PRD 本身就不完整，卻照著它寫了三天**」。模糊的 PRD 不會主動提醒你它模糊——它在你寫到第三個檔案、要拼一個你以為早就定義好的回傳格式時才爆。
- **選擇**：加一個 §2 Definition of Ready——動手寫第一行前，先健檢 PRD「能不能被實作」（逐端點 method/path/auth/schema/錯誤碼/冪等、逐表約束、認證歸屬…）。DoR 在入口、DoD 在出口，兩者對偶。
- **取捨**：動工前要先付一筆成本。但 DoR 明確**不是**「拒絕不完美的 PRD」——真實 PRD 幾乎都不完美。它只「辨識哪些不完美會擋路、哪些不會」：有 blocker 退回，有小洞標假設繼續。

### 決策 3：靠工具強制，不靠紀律

- **問題**：再多「紀律」「規則」「下次注意」，都擋不住趕工時的自我合理化。「12 個 endpoint 寫完 9 個就打勾」「commit 訊息寫『全部完成』但其實漏一半」——這些不是因為不懂規則，是因為**規則依賴人去記、去自律**。
- **選擇**：把判斷從「靠記憶」變成**機械化強制**——M-007 反聚合（checklist 禁止「12 個 ⬜」，一行一個 checkbox）、M-008 反向稽核（一支腳本比對 PRD vs code，**exit code 非 0 = 失敗**）、M-010 禁主觀詞（git hook 擋下「完整/done/全部」，逼你用「12/12 endpoints」）。
- **取捨**：你得真的去建、去跑這些檢查——存在卻不跑的檢查等於白建。但只要建了，它每次都會抓，不會因為你累了就放水。這是 Compass 最有原創價值的一條，貫穿它所有工具設計。

### 決策 4：腳本要 config-driven，不能寫死

- **問題**：原本的稽核腳本寫死了 FastAPI 的 `@app.get` pattern、特定的 PRD 章節編號。對一個專案能跑，對別人全錯。
- **選擇**：把腳本改成 **config-driven**——讀一個 `compass-audit.json`，裡面描述「你的 PRD 在哪、用什麼 regex 抓項目、掃哪些 code 檔」。腳本本身專案無關，每個專案自己寫 config。找不到 config 時印出範例並 exit 2（這是設計，不是故障）。
- **取捨**：每個專案要付一次寫 config 的成本。一個衍生教訓：這些印 emoji 的腳本在繁體中文 Windows console 會 `UnicodeEncodeError` 崩潰，而崩潰的 exit 1 會跟「PRD 有缺口」的業務碼撞號——靜默誤判。解法是腳本一律 `sys.stdout.reconfigure(encoding="utf-8")`。通用化不只是抽掉路徑，還要扛得住不同環境。

### 決策 5：「完成就是完成」——與敏捷和解

- **問題**：原 SOP 寫「一次寫完，不分階段，不切 V1/V2/V3」。但這跟業界主流（Agile / MVP / 漸進交付）直接衝突——漸進交付不是偷懶，是降低風險、拿真實回饋的正當手段。
- **選擇**：原 SOP 真正想說的不是「不准分階段」，而是「**不要留半成品**」。所以把規則軟化成「完成就是完成、不留半成品；可分小塊交付，但不可半成品階段化」。關鍵字從「階段」變成「**半成品**」。
- **取捨**：軟化後的規則更容易被鑽漏洞，所以反半成品的精神得靠別處強制（決策 3 的工具）。通用教訓：把「對我成立」變「對別人成立」時，要分清楚一條規則的**精神**和它的**字面**——字面可能跟別人的工作方式衝突，但精神往往是共通的。

### 決策 6：PRD 衝突分軌處理——靜態三類 + 動態三類

- **問題**：「PRD 跟我預期的不一樣」其實是好幾件不同的事，混在一起處理會災難。
- **選擇**：分流。**靜態三類**（PRD 不動，你讀它時發現）：*模糊* → 採具體一方，不停手；*bug* → 自相矛盾，停手等裁決；*缺漏 + 實作更好* → 6 道 gate 檢查，保留並等裁決。**動態三類**（原 SOP 沒涵蓋、但最高頻的真實衝突）：*PRD 中途變更*、*跨文件衝突*（按領域定權威）、*多 PRD 依賴*。
- **取捨**：要學的分支比「不確定就問」一條規則多。但精準分流讓你不會在模糊處浪費時間問人，也不會在真 bug 上硬幹。

### 決策 7：既有 code 是「第二份文件」

- **問題**：原 SOP 把 brownfield / 純 bug 修復列為「不適用」。但大部分真實工作就是 brownfield——這是個缺口。
- **選擇**：把既有 code 定位成 **PRD 之外的第二份真相來源**——它記錄了「系統現在實際怎麼跑」，其中可能藏著沒寫進 PRD 的隱性需求、邊界補丁、下游依賴。所以 PRD 跟既有 code 衝突是*跨文件衝突*（§5.3），不是「PRD bug」：按領域定權威，不要盲目照 PRD 覆寫。
- **取捨**：brownfield 工作會變慢、變謹慎。但風險不對稱——在既有 codebase 裡，弄壞既有行為的代價，遠高於漏做一個新功能。

### 決策 8：一份專門的 SCOPE——「說不」也是一種設計

- **問題**：Compass 的 scope 很容易模糊——「PRD 紀律」和「不是 PRD 紀律」的邊界很糊（PRD 撰寫？產品探索？PM 工具？架構決策？）。
- **選擇**：寫一份獨立的 [SCOPE](./SCOPE.md)，明文列出涵蓋與不涵蓋，並宣告整條工具鏈的分工。
- **取捨**：多一份要維護的文件，也縮小了被認知到的受眾。但一個工具想「什麼都做」，最後會「什麼都做不精」——明文的「不做」能防止誤用與期待錯位。

## 它刻意不做的事

以下是刻意畫下的邊界，不是缺漏。完整理由見 [SCOPE](./SCOPE.md)；精簡版：

- **它不是輕量提醒。** Compass 假設你願意花時間建追蹤文件、跑 audit、寫 checklist。想要輕量，請用 [Sentinel](https://github.com/RayLi-Git/sentinel)。
- **它不寫 PRD。** Compass 從一份已存在的 PRD 算起；把模糊想法逼成扎實規格是 [Cartographer](https://github.com/RayLi-Git/cartographer) 的工作。
- **它不假設有完美 PRD。** DoR 就是承認「真實 PRD 常常不完美」——所以開工前要先檢查 PRD 本身，而不是盲信。
- **它跟快速 MVP 文化有張力，這是刻意的。** 不是反對 Lean Startup——立場只是「當你決定要照 PRD 走時，就照 PRD 走完」。一句話：*PRD 不是 Compass 的敵人，PRD 中途改才是。*

---

> Compass 不是萬靈丹，是一把**特定情境下精準的工具**。它探索「如何把 PRD 驅動開發的紀律，編碼成 AI 寫程式搭檔的執行紀律」。其餘由工具鏈夥伴探索：[Cartographer](https://github.com/RayLi-Git/cartographer) 畫地圖、[Sentinel](https://github.com/RayLi-Git/sentinel) 管你怎麼想、[Lookout](https://github.com/RayLi-Git/lookout) 在桅杆瞭望。
