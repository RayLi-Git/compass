# §11.1 The 4 Tooling-Enforced Rules (M-007 ~ M-010)

> Part of [Compass](../../SKILL.md) §11 — Tooling-enforced discipline.
> Swap discipline that "relies on humans remembering" for discipline mechanically enforced by exit code / TodoWrite / commit-msg hook.

---

## Why these 4 rules exist

Proven: piling on "more rules + more discipline" has limited effect. **What actually prevents missed items is turning judgment from "relying on memory" into "mechanical enforcement."**

The 4 rules below are all tool-driven, not memory-driven. Each maps to a common vibe-coding "self-rationalized omission" scenario, and the fix is never "be more careful next time" — it's letting the tool block you.

---

## M-007: Completing a PRD enumerated list requires line-by-line matching (no aggregated check-off)

**Problem scenario**: The PRD's endpoint section lists 12 endpoints; the checklist collapses them into a single line "auth 12 ⬜". You write 9, mark it ✅, and the remaining 3 get buried. Aggregated entries are the biggest breeding ground for self-rationalized omissions.

**Rule**:

- Checklist item granularity that violates "1 PRD item = 1 checklist line" counts as a DoD failure.
- The following aggregation words are **banned** from the checklist: "N items", "multiple", "etc.", "the full set", "a series of".
- Must break down to the smallest checkable unit:
  - 1 endpoint = 1 line
  - 1 page / route = 1 line
  - 1 schema table = 1 line
  - 1 field constraint = 1 line
- **Tool enforcement**: use a script or LLM to automatically expand the PRD table line-by-line into checklist entries (avoids hand-copying that drops rows). See the example skeleton at [`/scripts/expand_checklist.example.py`](../../scripts/README.md).

**Pass conditions**:
- No aggregation words anywhere in the checklist.
- Every line maps to a specific PRD row / cell.
- Entry count ≥ the total row count of the PRD tables.

**Relationship to other sections**: detailed checklist granularity rules are in [§4 DoD](../04_quality_gates/01_dod.md); this rule is the tooled-up reinforcement of §4.

## M-008: DoD must run a reverse audit script

**Problem scenario**: the self-check loop only tests "does the feature I wrote run", not "did I write everything the PRD listed". This is the blind spot of "forward pass ≠ reverse coverage".

**Core insight**:

> **Rely on exit code, not on discipline.**

Discipline fatigues, and gets eroded by "the demo is right around the corner"; exit code does not. When the audit script exits non-zero, DoD is failed outright — you can't bypass it with "I feel it's close enough".

**Rule**:

- Before every delivery unit completes, run a reverse audit script (look up code from the PRD, not the PRD from code).
- **Non-zero exit code = DoD failure**. No subjective judgment like "I looked at the output and it seems OK".
- The audit script must at minimum cover:
  - Your PRD endpoint section (does each endpoint have a route definition in code)
  - Your PRD schema section (does each table / each field appear in a migration or model definition)
  - Your PRD page section (is each page / route registered in the frontend router)
  - Necessary critical-mechanism greps (e.g. PRD requires "JWT must verify expiry" → grep `exp` in the auth module)
- For PRD "descriptive requirements" (e.g. "should push a notification to subscribers"), audit cannot mechanically check it → rely on M-007's checklist granularity to catch it.

**Example skeleton**: a reference implementation is at [/scripts/audit_prd_vs_code.example.py](../../scripts/README.md) (config-driven, reads `compass-audit.json`). Every project should customize it to its own PRD structure and tech stack, but the "non-zero exit code = failure" rule never changes.

## M-009: TodoWrite mandatory

**Problem scenario**: tracking "what's left to do in this **unit**" from memory is unreliable. Across long conversations, sessions, and after context compaction, "I thought I'd already done it" is a common illusion.

> **Defining "unit"**: one unit = one PRD item / one module / one PR.
> Not an abstract phase, but the smallest delivery slice with a clear PRD mapping, a clear DoD, and that can be committed on its own.

**Rule**:

- Before starting each **unit**, use the TodoWrite tool to list the N todos mapping to this **unit**'s PRD.
  - Not abstract goals ("finish auth").
  - Concrete checkable items ("implement POST /login", "implement POST /logout", "JWT expiry verification").
- Mark each one `in_progress` → `completed` as you finish it.
- **No batch updates**: don't flip 5 items from pending to completed in one go. Update once per completed item, to avoid the "batch illusion".
- The system forcibly shows "N items still pending" — you can't claim "this **unit** is done" until N=0.
- Todo state is auto-saved before interruption; when picking up across sessions, read the todos before the progress doc.

**Pass conditions**:
- TodoWrite initialized at the start.
- Each completed item has a corresponding status change along the way.
- Pending count is 0 when claiming done.

## M-010: No subjective words in commit messages

**Problem scenario**: writing "auth module complete ✅ 255/255" manufactures a "done" illusion, but some PRD sub-section is half-missing. Subjective words are the final disguise of self-comfort.

**Rule — banned-word list**:

| Banned | Why banned |
|---|---|
| complete | unverifiable subjective assertion |
| all done | vague aggregation, hides omissions (bare "all" is too common, e.g. "all 12 endpoints" — deliberately not blocked to avoid false positives) |
| one-shot done | self-comfort phrasing |
| the full set | same as above |
| implemented | doesn't specify what was implemented |
| done | same as complete |
| finished | same as complete |

**Rule — must use concrete counts**:

- ✅ "12/12 endpoints pass audit"
- ✅ "3/9 templates, 6 missing — see progress.md"
- ✅ "auth module 5 endpoints done, audit 0 errors"
- ❌ "auth fully implemented ✅"
- ❌ "the full set of endpoints done"

**If the audit script shows missing items**: the commit message must explicitly list "⚠ Known missing: ..." + the corresponding todo recorded in the progress doc.

**Tooling**: lint subjective words with a commit-msg hook (example at [`/scripts/commit-msg-lint.sh`](../../scripts/README.md)). When the hook detects a banned word → it rejects the commit, no relying on a reviewer to catch it manually.

## Why these 4 rules differ from existing sections

| Dimension | §0–§10 existing | §11 (M-007 ~ M-010) |
|---|---|---|
| Depends on | discipline (humans remembering, humans looking) | tools (exit code / TodoWrite / commit-msg hook) |
| Failure mode | self-rationalize after skipping ("careful next time") | non-zero exit code blocks outright; hook rejects commit |
| Recoverability | one-off reminder, missed again next time | mechanical — caught on every re-run |
| Triggered by | you yourself (easily fatigued) | CI / pre-commit / DoD gate |
| Applies to | personal discipline | any collaborator (including future you) |

---

## Iron rule

> **If any of these 4 rules is broken and the tool didn't block it → the tool has a bug, and you must fix the tool, not just write "careful next time" in the log.**

This iron rule is the soul of all of §11.

"Careful next time" is a synonym for the failure mode:
- It dumps responsibility on future you.
- It attributes "didn't block the omission" to "the human wasn't diligent enough" rather than "the tool wasn't strict enough".
- It lets the same omission recur in the next **unit**.

The correct response:
1. **Find out why the tool didn't block it** (audit script doesn't cover this case? hook regex missed a word? checklist expansion dropped a row?).
2. **Fix the tool so the same class of error can't pass next time**.
3. Only then record the case in `.claude/debug-log.md` — but **the focus of the record is "what the tool was reinforced with"**, not "careful next time".
