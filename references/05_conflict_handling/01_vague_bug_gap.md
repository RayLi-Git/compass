# §5.1 PRD 模糊 / Bug / 缺漏：三類處置

> Part of [Compass](../../SKILL.md) §5 — Conflict Handling.
> 當 PRD 與現實對不上時，把「能解的」、「無解的」、「該補的」分流到三條不同的處置軌道，避免你在不該停的地方停、或在該停的地方硬幹。

---

## 為什麼要把 PRD 衝突分三類？

實務上「PRD 跟我預期的不一樣」其實是三件事，混在一起處理會災難：

- **模糊**：PRD 寫得不清楚，但兩種解讀其實可選一個繼續走 → 不該停。
- **Bug**：PRD 自相矛盾、邏輯錯誤、任一解讀都行不通 → 必須停，等使用者裁決。
- **缺漏 + 實作補上更好的**：PRD 沒寫，但你寫的時候自然冒出一個對齊設計原則的小補強 → 不該直接套 YAGNI 砍，也不能擅自擴張，要記錄並等裁決。

下面三節就是這三條軌道的 SOP。最後 §5.1.4 給你一張速查表。

> 📍 **本檔涵蓋「靜態 PRD」的三類衝突**（模糊 / bug / 缺漏）。PRD 中途變更、跨文件衝突（PRD vs ADR vs API contract）、多 PRD 依賴等「動態衝突」見 [§5.2](02_prd_change.md)、[§5.3](03_cross_document.md)、[§5.4](04_multi_prd.md)（皆已交付）。

---

## §5.1.1 PRD 模糊（寫得不清楚）

**症狀**：PRD 內部不一致、描述含糊、同一件事兩處寫法不同，但**任一解讀都能做、都不會炸**。

### 4 步處置

1. **記錄兩處原文**
   在開發紀錄檔（development log，見 [§3.2 追蹤文件](../03_implementation/02_tracking_docs.md)）的「PRD Ambiguity」區塊引用兩處原文，標明來源章節。

2. **依下列優先順序判斷採哪一邊**：
   1. **較具體、較可驗證的一方**（有明確輸入輸出、條件式、邊界值的）
   2. **較新章節**（PRD 後出現的通常為修訂）
   3. **與「設計原則 / 核心理念」章節一致的一方**
   4. **與術語表 / 名詞定義一致的一方**

3. **採用並繼續，不中斷詢問使用者**
   模糊不是 bug，自己拍板是哨兵應有的判斷力。

4. **在 PRD checklist 對應列加註「※ 採 §X.Y 解」**
   讓使用者事後 review 時一眼看到你的判斷點，能快速否決或追認。

### 關鍵心法

> 模糊不停手。停手是 §5.1.2 才做的事。
> 你選了之後在 checklist 留下足跡，使用者要回頭翻案隨時翻得到——這比每個模糊處都丟一個問題給使用者要省事得多。

---

## §5.1.2 PRD Bug（寫錯了）

**症狀**：發現的不是模糊，而是**邏輯錯誤**。典型例子：

- 權限矩陣自相矛盾，任一方解都會違反另一條規則
- API 簽章與 schema 衝突，無論照哪個寫都會 fail
- 狀態機某個 transition 不存在，但 PRD 又要求它要能跑
- 數學/業務邏輯計算公式錯誤，跟舉例對不上

判準：**任一解讀都行不通**——這就跟 §5.1.1 的「任一解讀都能做」對照而分。

### 6 步處置（停手 → 記錄 → 等裁決）

1. **立即停手，不繼續寫該模組**
   不要硬寫一個你知道是錯的東西。

2. **不自行「修正」PRD**
   PRD 是合約；你發現 bug 的責任是回報，不是改合約。改了等於把使用者的意圖偷偷替換成你的猜測。

3. **在開發紀錄檔（development log）的「PRD Issues」區塊記錄**（標 `[SKIPPED-PRD]` tag）：
   - 引用原文（哪幾段、哪幾節）
   - **為何認為是錯而非模糊**（這一點最重要，要說服自己也說服使用者）
   - 建議的 2–3 種解法，各自的代價

4. **跳過該模組，繼續其他不受影響的模組**
   不要讓一個 PRD bug 卡住整個 sprint。可平行進行的繼續做。

5. **在 progress.md 的待辦標「⚠ 等待 PRD 裁決」**
   讓使用者知道這條線是停在哪、為什麼停。

6. **等使用者裁決後再回填**
   裁決下來後，把該模組從 SKIPPED 狀態移出，按裁決寫，並把 debug log 那條標為 resolved。

### 關鍵心法

> 你不是 PRD 的作者，你是 PRD 的執行者。
> 看到 bug 第一反應如果是「我幫你改一下好了」，那是越權；正確反應是「停、log、等」。

---

## §5.1.3 PRD 缺漏（gap）+ 實作補上更好的

**症狀**：寫一寫，自然冒出一個 PRD **沒明列**、但**對齊設計原則**的合理小補強。

例：API response 多回了一個 `updated_at` 欄位讓前端能算 cache 失效；middleware 多塞一個 `X-Content-Type-Options: nosniff`；邊界輸入給了一個合理 fallback。

這類情境的陷阱是：**直接套 YAGNI 砍掉**會丟失真正有價值的補強；**直接保留不講**又會偷偷擴張 scope。所以走一條中間路線——**保留 + 標記 + 等裁決**。

### 5 步處置

1. **不立刻砍回**
   先承認這個小補強是有思考過的，不是隨手加的。

2. **在開發紀錄檔（development log）的「PRD Gaps」區塊記錄**：
   - 實作了什麼（具體欄位 / header / 行為）
   - PRD 缺漏的位置（§X.Y 應該寫但沒寫）
   - 為何較好（對齊哪條設計原則 / 安全鐵則 / 使用者體驗考量）
   - 不做的話會缺什麼（缺失的後果）

3. **在 progress.md 的待辦標「⚠ 等 PRD 補強裁決」**

4. **暫時保留實作，繼續其他模組**
   不要為了這個小補強卡住，但也不要假裝它不存在。

5. **等使用者裁決**：
   - **採納** → 由**使用者改 PRD**（你不自行改）→ PRD checklist 補一條 → 維持實作
   - **不採納** → 砍回 PRD 規格 → 在 debug log 記錄被砍的理由（這對下次同類判斷有幫助）

### 判定門檻（任一不符 → scope creep，直接砍，不走 §5.1.3）

進入這條軌道前，先用下面這 6 個閘門檢查。**任一不符 → 不適用 §5.1.3，直接砍回 PRD 規格**：

- [ ] 對齊 PRD 的「設計原則 / 核心理念」章節
- [ ] 不違反 PRD 的「安全」鐵則（特別是 PII 隔離、權限邊界）
- [ ] 不引入 PRD「技術選型」章節外的新依賴
- [ ] 不擴大資料模型（不新增表、不新增欄位）
- [ ] 不影響其他模組的 API contract（path / method / 回傳結構）
- [ ] 新邏輯 ≤ 約 20 行（超過代表已是新功能——不是 bug 也不是小補強，應停手把它當「PRD 缺漏的大型增強」回報使用者，等裁決是否納入 PRD，而非自行寫下去）

### 9 種典型情境對照

| 情境 | 處置 |
|---|---|
| API 多回 PRD 沒寫但顯然該回的欄位（例 `updated_at`）| 走 §5.1.3，可暫留 |
| Middleware 多塞安全 header（例 `nosniff` / CSP 基礎防線）| 走 §5.1.3，可暫留 |
| 邊界輸入 fallback（例空字串→400，避免下游 NPE）| 走 §5.1.3，可暫留 |
| 錯誤情境 log 補強（讓未來 debug 不抓瞎）| 走 §5.1.3，可暫留 |
| 新增 PRD 沒列的 endpoint / 模組 | **必砍**（這是新功能不是補強） |
| 加表 / 加欄位 | **必砍**（資料模型擴張，要裁決） |
| 引入新套件 | **必砍**（技術選型擴張） |
| 改變既有 API shape | **必砍**（破壞 contract） |
| 過度抽象（看到兩個案例就寫 strategy / factory）| **必砍**（YAGNI 範疇） |

### 關鍵心法

> §5.1.3 是給「克制過後仍覺得該補」的小東西用的，不是「我手癢想多寫」的免死金牌。
> 6 個閘門過不了 → 那不是補強，是 scope creep，請走 YAGNI 直接砍。

## §5.1.4 三類處置分工速查

碰到 PRD 跟你預期對不上時，先用這張表分流：

| 情境 | 走哪節 | 是否停手 | 動作關鍵字 |
|---|---|---|---|
| 模糊 / 內部不一致，但**任一解讀都能繼續做** | §5.1.1 | **不停** | 判斷準則 → 採用 → checklist 加註 |
| 邏輯錯誤、自相矛盾，**任一解讀都做不出來** | §5.1.2 | **停手等裁決** | log `[SKIPPED-PRD]` → 跳過模組 → 等 |
| PRD 沒寫，但實作時自然補上一個**對齊原則的小補強** | §5.1.3 | **不停，但保留並等裁決** | 6 閘門檢查 → log Gap → 等 |
| PRD 沒寫，**也沒實作上的必要** | YAGNI（見 [§1 核心原則](../01_foundations/01_principles.md) 第 8 誡） | — | 不寫 |

### 三類處置共同的記錄習慣

無論走哪一節，三件事一定要做：

1. **開發紀錄檔（development log）留痕**（PRD Issues / PRD Gaps / PRD Ambiguity 各自分區）
2. **progress.md 待辦標記**（⚠ 等裁決 / ※ 採 §X.Y 解 / ⚠ 等補強裁決）
3. **裁決下來後**，把對應條目從待裁決狀態 resolve 掉，並把 PRD checklist 同步更新

少做任何一件，下次 session 開始時你會搞不清楚哪些是已決、哪些還在等——這比 PRD 本身的問題還麻煩。
