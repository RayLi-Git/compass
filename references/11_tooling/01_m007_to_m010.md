# §11.1 工具強制 4 條 (M-007 ~ M-010)

> Part of [Compass](../../SKILL.md) §11 — Tooling-enforced discipline.
> 把「靠人記得」的紀律，換成「靠 exit code / TodoWrite / commit-msg hook」的機械化強制。

---

## 為什麼需要這 4 條

經驗證：靠「再多規則 + 紀律」效果有限。**真正能防漏項的是把判斷從「靠記憶」變成「機械化強制」**。

以下 4 條規則均為工具導向，不靠人記。每一條都對應一個 vibe coding 常見的「自我合理化漏項」場景，而解法都不是「下次注意」，而是讓工具替你擋住。

---

## M-007：完成 PRD 明列清單時必須逐行對照（不可聚合打勾）

**問題情境**：PRD 的 endpoint 章節列出 12 個 endpoint，checklist 寫成「auth 12 個 ⬜」一條，寫完 9 個就打 ✅，剩 3 個被掩蓋。聚合條目是「自我合理化漏項」的最大溫床。

**規則**：

- checklist 條目顆粒度違反「1 PRD 項 = 1 checklist 條」即視為 DoD 失敗。
- 以下聚合詞**禁止**出現在 checklist：「N 個」「多個」「等等」「全套」「一系列」。
- 必須拆到最小可勾選單位：
  - 1 個 endpoint = 1 條
  - 1 個 page / route = 1 條
  - 1 個 schema 表 = 1 條
  - 1 個欄位約束 = 1 條
- **工具強制**：使用一個 script 或 LLM 自動把 PRD 表格逐行展開為 checklist 條目（避免人手抄漏行）。範例骨架見 [`/scripts/expand_checklist.example.py`](../../scripts/README.md)。

**通過條件**：
- checklist 中沒有任何聚合詞。
- 每一條都能對應到 PRD 的具體某一行 / 某一格。
- 條目數 ≥ PRD 表格的行數總和。

**與其他章節的關係**：詳細 checklist 顆粒度規則見 [§4 DoD](../04_quality_gates/01_dod.md)；本條是 §4 的工具化補強。

## M-008：DoD 必跑反向審計 Script

**問題情境**：自檢循環只測「我寫的功能能不能跑」，沒測「PRD 列的東西我有沒有寫」。這是「正向通過 ≠ 反向覆蓋」的盲點。

**核心洞察**：

> **靠 exit code，不靠紀律。**

紀律會疲勞、會被「demo 在即」侵蝕；exit code 不會。當 audit script 退出碼非 0，DoD 直接判失敗——你無法用「我覺得差不多了」繞過去。

**規則**：

- 每個交付單元完成前必跑反向審計 script（從 PRD 反查 code，而非從 code 反查 PRD）。
- **退出碼非 0 = DoD 失敗**。不接受「我看了 output 覺得 OK」這種主觀判斷。
- audit script 至少要覆蓋：
  - 你 PRD 的 endpoint section（每個 endpoint 在 code 裡是否存在 route 定義）
  - 你 PRD 的 schema section（每張表 / 每個欄位是否在 migration 或 model 定義中出現）
  - 你 PRD 的 page section（每個 page / route 是否在前端路由中註冊）
  - 必要的關鍵機制 grep（例如 PRD 要求「JWT 必須驗 expiry」→ grep `exp` 在 auth 模組）
- 對 PRD「描述式要求」（如「應推送通知給訂閱者」）audit 無法機械檢出 → 仰賴 M-007 的 checklist 顆粒度兜住。

**範例骨架**：示範實作見 [/scripts/audit_prd_vs_code.example.py](../../scripts/README.md)（config-driven，讀 `compass-audit.json`）。每個專案都應該根據自己的 PRD 結構與技術棧客製，但「exit code 非 0 = 失敗」這條不變。

## M-009：TodoWrite 強制使用

**問題情境**：靠記憶追蹤「這個**單元 (unit)** 還剩什麼沒做」，記憶不可靠。長對話、跨 session、context 壓縮過後，「我以為已經做完」是常見錯覺。

> **定義「單元 (unit)」**：一個 unit = 一個 PRD 項目 / 一個模組 / 一個 PR。
> 不是抽象階段，而是有明確 PRD 對應、有明確 DoD、可以單獨 commit 的最小交付塊。

**規則**：

- 每個**單元 (unit)** 開工前用 TodoWrite 工具列出本**單元 (unit)** PRD 對應的 N 項 todo。
  - 不是抽象目標（「完成 auth」）。
  - 是具體可勾選項（「實作 POST /login」「實作 POST /logout」「JWT expiry 驗證」）。
- 每完成一個 mark `in_progress` → `completed`。
- **不准批次更新**：不可一次把 5 項從 pending 改成 completed。每完成一項就更新一次，避免「批次假象」。
- 系統強制顯示「還剩 N 項 pending」— 不可宣稱「本**單元 (unit)** 完成」直到 N=0。
- 中斷前 todo 狀態自動保存，跨 session 接手時先讀 todo 再讀 progress 文件。

**通過條件**：
- 開工時有 TodoWrite 初始化。
- 過程中每完成一項都有對應的 status 變更。
- 宣稱完成時 pending 計數為 0。

## M-010：Commit Message 禁主觀詞

**問題情境**：寫「auth 模組完整 ✅ 255/255」製造「已完成」假象，但 PRD 的某個 sub-section 漏一半。主觀詞是「自我安慰」的最後一道偽裝。

**規則 — 禁用詞清單**：

| 禁用 | 為什麼禁 |
|---|---|
| 完整 / complete | 無法驗證的主觀斷言 |
| 全部 / all done | 模糊聚合，掩蓋漏項（裸字 all 太常見如 "all 12 endpoints"，刻意不擋以免誤判）|
| 一次到位 | 自我安慰用詞 |
| 全套 | 同上 |
| 已實作 | 不指明實作了什麼 |
| done | 同 complete |
| finished | 同 complete |

**規則 — 必用具體計數**：

- ✅「12/12 endpoints 通過 audit」
- ✅「3/9 templates，漏 6 項見 progress.md」
- ✅「auth 模組 5 endpoints 完成，audit 0 errors」
- ❌「auth 完整實作 ✅」
- ❌「全套 endpoints done」

**若 audit script 顯示漏項**：commit message 必須明列「⚠ 已知漏項：...」+ 對應的待辦記錄在 progress 文件。

**工具化**：用 commit-msg hook lint 主觀詞（範例見 [`/scripts/commit-msg-lint.sh`](../../scripts/README.md)）。hook 偵測到禁用詞 → 拒絕 commit，不靠 reviewer 手動抓。

## 為什麼這 4 條與既有章節不同

| 維度 | §0–§10 既有 | §11 (M-007 ~ M-010) |
|---|---|---|
| 依賴 | 紀律（人去記、人去看） | 工具（exit code / TodoWrite / commit-msg hook） |
| 失敗模式 | 跳過後自我合理化（「下次注意」） | exit code 非 0 直接擋；hook 拒絕 commit |
| 復原性 | 一次性提醒，下次照樣漏 | 機械化每次重跑都會抓 |
| 觸發者 | 你自己（容易疲勞） | CI / pre-commit / DoD gate |
| 適用對象 | 個人紀律 | 任何協作者（包含未來的你） |

## 鐵律

> **若這 4 條任一條被破而沒被工具擋下 → 工具有 bug，必須修工具，不准只在 log 寫「下次注意」。**

這條鐵律是整個 §11 的靈魂。

「下次注意」是失敗模式的代名詞：
- 它把責任丟給未來的你。
- 它把「沒擋下漏項」這件事歸因到「人不夠認真」，而不是「工具不夠嚴」。
- 它讓同樣的漏項在下一個**單元 (unit)** 重演。

正確的反應：
1. **找出工具為什麼沒擋下**（audit script 沒覆蓋這個 case？hook regex 漏字？checklist 展開漏行？）
2. **修工具，讓下次同類錯誤無法通過**。
3. 才在 `.claude/debug-log.md` 記錄案例 — 但**記錄的重點是「工具被補強了什麼」**，不是「下次注意」。
